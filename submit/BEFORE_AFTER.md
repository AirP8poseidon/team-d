# Before / After — 수작업 → 에이전트화

## 우리가 자동화한 반복 수작업
사내 HPC 사용자가 **노드 사용 현황·충돌, 서버 환경 정보, 사용 노하우를 메신저/구두/개별 문의로 매번 확인·조율**하던 것 → **사내 HPC 통합 포털**(모니터링·정보공유·실시간 채팅·사용량 통계·시스템 상태)로 한곳에 모은다.

> 본 문서는 **(1) 업무 자동화 효과**와 **(2) 개발 방식(설계·협업) 효과** 두 층을 함께 기록한다. (1)의 정량치는 구현·시연 후 채움.

## Before (기존 수작업 절차)
1. 노드가 비었는지/누가 쓰는지 메신저로 묻거나 직접 ssh로 확인 → 중복 작업·충돌 발생.
2. 서버 환경(모듈·드라이버·sbatch 사용법)을 매번 사람에게 물어봄.
3. 온도·디스크·NFS 같은 운영 상태는 관리자만 알고 사용자에겐 안 보임.
4. (개발 측면) 협업 설계는 구두 합의 + 흩어진 메모, 새 세션마다 맥락 재설명.

## After (에이전트화된 절차)
1. 포털 한 화면에서 노드별 사용률 + **"누가·왜 쓰는 중"** 맥락 + 시스템 상태를 즉시 확인.
2. 어느 페이지에서든 **플로팅 채팅**으로 실시간 소통, 위키에서 환경·사용법 셀프 확인.
3. (개발 측면) **인터뷰 1회 → 청사진 1문서(DEV_BLUEPRINT) → 팀원별 핸드오버·킥오프 프롬프트** 파생 → 받아서 바로 병렬 착수.

## 효과 (해당 지표만)

| 지표 | Before | After | 효과 |
|------|--------|-------|------|
| 협업 설계 산출물 | 구두 합의·흩어진 메모 | 청사진 1문서(역할·DB·API·통합 접점) | 새 세션·새 팀원도 문서 하나로 착수 |
| 팀원 착수 준비 | 말로 전달, 경계 모호 | 자기완결 핸드오버 2종 + 붙여넣기 킥오프 프롬프트 | 받는 즉시 자기 파트 시작, 질문 왕복 ↓ |
| 파일 충돌 위험 | 동시 수정 우려 | 소유 파일 분리 명문화 | 충돌 0 설계, 통합은 "모으기 + 접점 1개" |
| 노드 현황 파악(업무) | 메신저/구두/ssh 개별 확인 | 포털 한 화면 통합 | (정량치 구현 후 측정) |

> `PROCESS_LOG.md`의 효과 측정과 같은 지표로 적음.
